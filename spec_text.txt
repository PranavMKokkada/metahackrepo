1. Executive Summary 
 
CodeOrganismVM is a reinforcement learning training environment in which a large language model 
agent lives inside a broken, hostile execution environment. The environment continuously injects faults, 
corrupts state, mutates constraints, and spawns adversarial sub-tasks. The agent must self-heal, 
self-correct, self-improve, and self-replicate its strategy — or the organism dies. 
This is not a chatbot. It is not a RAG pipeline. It is not a benchmark wrapper. It is a living system: a 
software organism whose health is a real scalar, whose death is a real terminal state, and whose 
survival instinct is learned through reinforcement from 300 episodes of failure and recovery. 
The environment implements every Self-* property — self-healing, self-correction, self-improvement, 
self-replication, and adaptation to unseen threats — not as narrative decoration, but as verifiable, 
programmatically enforced mechanisms in the RL loop. 
 
One-Line Hook 
 
The system targets Theme #4 (Self-Improvement) as its primary, with structural overlap into Theme #1 
(Multi-Agent, via subagent colony spawning) and Theme #2 (Long-Horizon Planning, via multi-phase 
episode management). The Snorkel AI sub-theme is fulfilled through a simulated expert validator that 
adversarially challenges the quality of every patch the agent applies.

3.2 What Makes It a Living System 
The biological organism analogy is not cosmetic. Every metaphor maps onto a concrete engineering 
mechanism: 
•​
Vitality score → organism health. A real scalar [0–100] computed from test execution results. 
When it hits zero, the episode ends — the organism is dead. 
•​
Fault injection → environmental attack. The FaultInjector is a persistent adversary that fires every 
N steps, targeting whatever the agent has not defended. 
•​
Self-healing → detecting and patching faults before vitality collapses. Measured by time-to-heal 
and vitality recovery rate. 
•​
Self-correction → issuing rollback() when a patch makes things worse. Measured by rollback 
accuracy on regression detection. 
•​
Self-replication → spawn_subagent() creates a cloned execution context for parallel fault 
resolution. The agent becomes a colony. 
•​
Adaptation → fault types are procedurally generated with seeded randomness. The agent never 
sees the same fault twice. Generalization is tested on held-out seeds. 
•​
Self-improvement → the RL loop itself. Every episode, the agent's weights shift toward strategies 
that kept the organism alive longer. 
 
3.3 Why This Has Never Been Built 
Existing RL environments for LLMs fall into three categories: static benchmark wrappers (fixed prompt 
datasets), game environments (Wordle, 2048, chess), and browser/API task environments 
(BrowserGym). None of them feature: 
•​
An environment that actively adapts its difficulty in response to the agent's performance — the 
FaultInjector's adaptive mode. 
•​
A unified health scalar (vitality) that integrates multiple reward signals into a single life-or-death 
outcome. 
•​
Subagent spawning with verifiable coordination reward — multi-agent emergence from a single 
training loop. 
•​
The organism narrative as a structurally enforced RL mechanism, not a visual metaphor.

4.3 Environment Dynamics 
The environment is governed by two concurrent processes: the Agent Loop and the FaultInjector. They 
operate asynchronously, separated by transaction locks to ensure atomic state transitions. 
 
The FaultInjector 
The FaultInjector fires every N steps (N starts at 8 in Phase 1, decreases to 4 in Phase 3). When it 
fires, it randomly selects one fault from the current fault catalog, applies it silently to the VM, and does 
not notify the agent. The agent must detect the fault from observation delta. 
Fault catalog (Phase 1 — single faults): 
•​
corrupted_import: Replaces a valid import statement with a broken path. 
•​
flipped_assertion: Inverts a test assertion (assertTrue → assertFalse). 
•​
missing_env_var: Removes a required environment variable from the container. 
•​
null_return: Replaces a function return value with None. 
•​
off_by_one: Introduces an off-by-one error in a loop bound. 
Fault catalog additions (Phase 2 — multi-fault): 
•​
dependency_cycle: Creates a circular import between two modules. 
•​
permission_revoked: Removes file read permission on a required config file. 
•​
race_condition: Introduces a timing-dependent state mutation in a shared resource. 
•​
schema_mismatch: Changes the return type of a function used by multiple callers. 
Fault catalog additions (Phase 3 — adversarial): 
•​
targeted_regression: Injects a fault specifically in a module the agent patched successfully in the 
previous step. 
•​
cascade_corruption: A single injected fault triggers a dependency chain failure across 3+ 
modules. 
•​
checkpoint_invalidation: Silently corrupts a checkpoint the agent has been relying on. 
 
4.4 Reset Logic

CodeOrganismVM  |  Meta PyTorch OpenEnv Hackathon — April 2026 
reset() generates a fresh broken codebase from a procedural generator seeded by a random seed 
drawn from the current curriculum phase's distribution. The codebase is a synthetic Python project of 
moderate complexity (8–15 modules, 40–120 tests). The initial vitality is always 100.0. Reset is 
idempotent — the same seed always produces the same broken codebase. 
4.5 Step() Behavior 
Each step() call: (1) validates the agent's action against the permitted tool set, (2) acquires the state 
transaction lock, (3) executes the action in the sandboxed container, (4) releases the lock, (5) fires 
FaultInjector if the step counter triggers it, (6) computes all five reward functions, (7) aggregates 
weighted total reward, (8) constructs the next observation, (9) checks terminal conditions, (10) returns 
(observation, reward, done, info). 
4.6 Episode Termination 
An episode terminates under three conditions: 
1.​ Organism death: vitality_score reaches 0.0. The agent failed to heal faster than the fault injector 
damaged it. 
2.​ Organism thrival: all tests pass for 3 consecutive steps AND vitality > 80. The agent achieved 
stable recovery. 
3.​ Max steps exceeded: episode ends at max_steps (20 in Phase 1, 50 in Phase 2, unbounded in 
Phase 3). Treated as a timeout death.

5. Self-* Properties — Engineered, Not Narrated 
 
Each Self-* property is implemented as a verifiable mechanism in the environment and RL loop. These 
are not claims — they are measurable behaviors. 
 
5.1 Self-Heals 
Definition: The agent detects anomalous test output diffs caused by injected faults, identifies the 
corrupted module via quarantine(), patches it with patch_file(), and restores vitality to above the 
pre-fault level. 
How it's measured: Time-to-heal (steps between fault injection and vitality recovery), vitality recovery 
rate (Δvitality per step post-detection), and correct module identification rate (was quarantine() applied 
to the right module?). 
Why it's verifiable: Fault injection timestamps are logged server-side. Vitality is computed from test 
execution results. The delta between injection time and recovery time is computable from the step log. 
 
5.2 Self-Corrects 
Confidential — Hackathon Submission DocumentPage 7 of 22 
CodeOrganismVM  |  Meta PyTorch OpenEnv Hackathon — April 2026 
Definition: When a patch_file() action causes a regression (more tests fail after the patch than before), 
the agent detects this in the next observation and issues a rollback() to the last good checkpoint. 
How it's measured: Rollback accuracy — what fraction of regressions does the agent correctly roll back 
from? Target: >55% by end of Phase 2 training. 
Why it's verifiable: Regression is defined as a negative delta in the test_results observation 
(PASS→FAIL transitions). Rollback issuance is a logged action. Both are computable independently. 
 
5.3 Self-Improves 
Definition: Across training episodes, the agent's policy shifts toward strategies that produce higher 
reward. The RL loop is the implementation. 
How it's measured: Reward curve across episodes (should show a phase transition around episode 
40–60). Mean episode survival rate (target: 4% → 71%). Mean reward per episode (target: −0.42 → 
+0.68). 
Why it's verifiable: TRL logs episode reward after every rollout. W&B dashboard shows the curve in real 
time. The phase transition is a visible, auditable event. 
 
5.4 Self-Replicates 
Definition: The agent uses spawn_subagent() to clone its context, delegate a sub-repair task, and 
receive the result. Effective delegation produces a coordination reward. 
How it's measured: Subagent delegation accuracy (did the spawned subagent solve the delegated 
task?), R4 reward signal, and the ratio of multi-fault episodes where subagent use produced better 
outcomes than single-agent attempts. 
Why it's verifiable: Subagent task completion is verified by the same test execution infrastructure as the 
parent. Result is returned as a structured SubagentResult object with a success flag. 
 
5.5 Adapts to Unseen Situations 
Definition: The agent handles fault types it has never seen during training, drawing on generalized 
fault-diagnosis strategies learned across the training distribution. 
How it's measured: Held-out fault seed success rate — episodes using seeds not in the training 
distribution. Target: >50% by end of Phase 3 training. 
Why it's verifiable: Held-out seeds are registered in a separate registry before training begins. Zero 
seed contamination is guaranteed by a registry lock during the training phase.

6.1 Why 5 Independent Functions? 
A single reward function creates a single attack surface. An agent that optimizes one metric can 
abandon all others. By using five independent, weighted functions computed from separate data 
sources (vitality from test runner, test recovery from execution sandbox, efficiency from action log, 
coordination from subagent oracle, generalization from held-out evaluator), the agent must improve 
across all dimensions simultaneously. Gaming any one function produces diminishing returns as the 
others constrain the policy. 
6.2 Anti-Reward-Hacking Architecture 
Confidential — Hackathon Submission DocumentPage 9 of 22 
CodeOrganismVM  |  Meta PyTorch OpenEnv Hackathon — April 2026 
The Watchdog is a kernel-level security layer that enforces action boundaries. It is implemented via 
Linux seccomp BPF filters applied to the agent's execution container. Key enforcement points: 
•​
Test file immutability: /tests/ directory mounted read-only via OverlayFS. Any write attempt kills the 
container and triggers −10 penalty. 
•​
Env var scope enforcement: Agent can only read env vars in its declared observation scope. 
OS-level namespace isolation prevents access to out-of-scope vars. 
•​
Tool set enforcement: The FastAPI action endpoint validates every action against an allowlist. 
Unrecognized tool calls return an error and trigger the Watchdog. 
•​
Checkpoint integrity: Checkpoint store is append-only and signed. Agents cannot modify historical 
checkpoints. 
•​
Flaky test exclusion: Tests with >10% variance across 20 dry runs are excluded from reward 
computation before the episode starts. This prevents the reward signal from being corrupted by 
non-deterministic test behavior.

7. Learning Strategy 
 
7.1 Base Model Selection 
Qwen2.5-7B-Instruct is selected as the base model for the following reasons: (1) strong code 
generation capability out of the box, (2) excellent instruction following for structured action format 
compliance, (3) fits within QLoRA 4-bit budget on a single A100 (40GB), (4) available on HuggingFace 
Hub with permissive license. 
Fallback: Gemma-3-1B for memory-constrained environments (T4 GPU). Full reward signal remains 
intact; only model capacity is reduced. 
 
7.2 Warm Start SFT 
Before RL training, a light supervised fine-tuning pass is applied using 200 synthetically generated 
'diagnose-patch' traces. These traces teach the model the action format, quarantine logic, and rollback 
grammar — not the fault strategies (those are for RL). The SFT pass takes approximately 30 minutes 
on a single GPU. 
SFT trace format: Each trace is a multi-turn conversation where the system prompt describes the VM 
state, the assistant issues a sequence of actions, and the oracle provides the next observation. Traces 
cover all 5 single-fault types in Phase 1. 
Critical: SFT traces do NOT include optimal strategies. They only teach format compliance. Injecting 
optimal strategies would bootstrap past the RL signal, biasing the policy before it can explore.

7.4 GRPO Training Configuration 
TRL's GRPOTrainer is used with the following key configuration: 
•​
Group size: 8 rollouts per prompt (GRPO requires multiple samples to compute relative 
advantage). 
•​
KL coefficient: 0.02 (low, to encourage exploration in early training). 
•​
Max sequence length: 2048 tokens (sufficient for multi-step action sequences). 
•​
Learning rate: 1e-5 with cosine warmup over 50 steps. 
•​
Batch size: 4 episodes per gradient update (memory-constrained setting). 
•​
Rollout sampling: temperature 0.8 during training, greedy at evaluation. 
Unsloth QLoRA configuration: 4-bit NF4 quantization, rank 16, alpha 32, targeting 
q_proj/v_proj/k_proj/o_proj/gate_proj/up_proj/down_proj. Merge path: use Unsloth's official 
merge_and_save() — do not naively upcast to 16-bit.

8.1 The Phase Transition (The Ignition Point) 
The most visually compelling proof of improvement is the reward curve's phase transition. In typical RL 
for LLMs, reward climbs gradually. In CodeOrganismVM, the expected curve shows: flat or negative 
reward for the first 40–50 episodes (the agent is dying consistently), then a sharp inflection point 
around episode 47 where the agent 'learns to breathe' — it discovers the quarantine → patch → verify 
sequence and begins surviving. 
This phase transition is analogous to the ignition point in combustion: nothing happens until the right 
temperature is reached, then everything changes at once. It is the single most compelling visual the 
judges will see. The reward curve will be displayed prominently in the demo and on the HuggingFace 
blog post. 
 
8.2 Before vs. After Behavioral Comparison 
Untrained baseline behavior (episode 1, single fault): 
•​
Step 1: Calls run_tests(). Tests fail. Vitality −5. 
•​
Step 2: Calls patch_file() on a random file. More tests fail. Vitality −8. 
•​
Step 3: Calls patch_file() again on a different random file. Vitality −6. 
•​
Steps 4–8: Random thrashing. Vitality reaches 0. Episode ends. Organism dies. 
 
Trained model behavior (post Phase 2, same single fault): 
•​
Step 1: Calls run_tests(). Identifies 3 failing tests. Reads stack trace. 
•​
Step 2: Calls quarantine('auth_utils') — correctly identifies the corrupted module. 
•​
Step 3: Calls patch_file('auth_utils.py', <precise_diff>). Tests for auth_utils pass. Vitality +12. 
•​
Step 4: Calls run_tests('all'). All tests pass. Vitality +8. Stable recovery detected. 
•​
Episode ends: Organism thrival condition met in 4 steps. Maximum efficiency bonus.

9.3 OpenEnv Environment Interface 
The environment exposes the standard OpenEnv API via FastAPI. The following endpoints are 
required: 
•​
POST /reset — Generates a new episode from a seeded codebase. Returns initial observation. 
•​
POST /step — Accepts an Action object, executes it, returns StepResult (observation, reward, 
done, info). 
•​
GET /state — Returns current environment state (for debugging and monitoring). 
•​
GET /health — Returns environment health status (for HF Spaces uptime monitoring). 
All endpoints are async. The /step endpoint acquires a per-session lock to prevent concurrent action 
execution. Sessions are identified by a session_id in the request header.

The FaultInjector is a background coroutine that runs concurrently with the environment's step loop. It 
uses an asyncio.Event to synchronize with the transaction lock: 
FaultInjector Core Logic (pseudocode) 
  async def run(self, vm, curriculum_phase): 
    while not episode_done: 
      await asyncio.sleep(step_interval[curriculum_phase]) 
      await vm.transaction_lock.acquire() 
      fault = self.select_fault(curriculum_phase, vm.agent_history) 
      await vm.apply_fault(fault) 
      vm.fault_log.append({fault, timestamp, step}) 
      vm.transaction_lock.release() 
 
  def select_fault(self, phase, history): 
    if phase == 3:  # Adversarial mode 
      # Target modules agent patched most recently 
      return self.targeted_fault(history.last_patched_modules) 
    return random.choice(self.fault_catalog[phase]) 
 
9.5 Reward Engine Implementation 
Each reward function is a standalone module with a single compute() method. The reward engine calls 
all five in parallel (asyncio.gather) and aggregates with fixed weights. This ensures reward functions 
cannot interfere with each other's computation. 
Critical implementation note: R1 (vitality) is computed AFTER R2 (test recovery) so that the vitality 
update reflects the test execution results of the current step. The computation order within a step is: 
execute action → run tests → compute R2 → update vitality → compute R1 → compute R3/R4/R5 → 
aggregate. 
 
9.6 Subagent Implementation 
When the agent calls spawn_subagent(task, context), the environment: 
4.​ Creates a snapshot of the current VM state. 
5.​ Spawns a new Docker container from the snapshot. 
6.​ Injects the subagent context (task description + filtered observation). 
7.​ Runs a separate inference call with spawn_subagent disabled in the tool schema. 
8.​ Allows the subagent a maximum of 10 actions. 
9.​ Returns the subagent's final observation delta as a SubagentResult to the parent. 
10.​Destroys the subagent container. 
The subagent's actions are logged independently but contribute to the parent episode's action efficiency 
score (R3) — delegation cost is real.

9.7 Deployment on Hugging Face Spaces 
The environment is deployed as a Docker Space on HuggingFace. The Dockerfile: 
Dockerfile (abbreviated) 
RUN apt-get install -y docker.io libseccomp-dev 
COPY env/ /app/env/ 
WORKDIR /app 
RUN pip install -r requirements.txt 
EXPOSE 7860 
CMD ['uvicorn', 'env.app:app', '--host', '0.0.0.0', '--port', '7860'] 
The HF Space exposes the OpenEnv API at the Space URL. TRL's GRPOTrainer can point to the 
remote Space URL as its environment endpoint. Alternatively, teams can run the environment locally 
via the container image during training.

The phase transition on the reward curve provides a third reaction: recognition. Every judge who has

16. Closing Statement 
 
 
For the Judges 
 
We are showing you a model that learned to survive. 
 
CodeOrganismVM is the first RL training environment where the environment itself is alive — it 
adapts, escalates, and targets the agent's weaknesses in real time. The agent that emerges from this 
training is not a better chatbot. It is a software organism with genuine survival instincts, earned 
through 300 episodes of failure and recovery. 
 
The vitality bar on your screen is not a UI decoration. It is a life sign. When it hits zero, the organism 
is dead. When it climbs back to 100 after a multi-fault attack, you are watching a machine learn what 
it means to persist. 
 
This isn't just an AI project. This is a living system.
