# Hackathon Winning Roadmap

This document is a blunt, implementation-focused roadmap for turning the current Autonomous SRE OpenEnv project from a strong prototype into a serious hackathon contender.

The current project has a real environment, tests, API surface, Docker support, and a compelling idea. The missing piece is not more branding. The missing piece is evidence: real training, honest improvement, runnable submission artifacts, and a story that judges can verify quickly.

## Executive Verdict

Current estimated readiness:

| Target | Readiness |
|---|---:|
| Minimum viable submission | 60-65% |
| Strong finalist submission | 55-60% |
| Winning submission | 45-50% |

The project can become competitive if the remaining work focuses on:

1. Real training or at least real environment-connected rollout evaluation.
2. Baseline-vs-trained or baseline-vs-improved-agent evidence.
3. Committed plots and machine-readable results.
4. A corrected README that proves claims instead of only stating them.
5. A working Hugging Face Space and short demo material.
6. Stronger reward anti-gaming and more realistic SRE incident tasks.

## What Judges Care About

From the OpenEnv Hackathon opening ceremony material, the judging weights are:

| Criterion | Weight | What It Means |
|---|---:|---|
| Environment Innovation | 40% | Is the environment novel, challenging, and useful for teaching an LLM a meaningful skill? |
| Storytelling & Presentation | 30% | Can judges quickly understand the problem, environment, learned behavior, and demo? |
| Showing Improvement in Rewards | 20% | Is there observable evidence that training or policy improvement worked? |
| Reward & Training Pipeline | 10% | Is the reward coherent, trainable, and connected to actual behavior? |

Minimum requirements include:

- Use OpenEnv/latest framework conventions.
- Provide a working training script using Unsloth or Hugging Face TRL.
- Provide evidence of actual training: loss/reward plots from a real run.
- Provide README links to HF Space and supporting material.
- Push a runnable environment to Hugging Face Space.
- Explain the problem, environment, and results clearly.

## Highest-Risk Gaps

These are the gaps most likely to hurt judging.

### 1. Training Evidence Is Currently Weak

Current state:

- `training/grpo_train.py` is a placeholder-style script.
- Reward functions return constants.
- The notebook contains mock reward curve data.
- README claims a 60% MTTR reduction without committed proof.
- No real trained model, LoRA artifact, W&B run, Trackio dashboard, or metrics JSON is committed.

Why this matters:

Judges explicitly said a messy but ambitious environment with real training evidence beats a polished but boring one. Right now, the project is ambitious and polished, but the proof is thin.

Required work:

- Implement an actual environment-connected training/evaluation loop.
- Save real rollout logs.
- Generate real plots from logged rewards and scores.
- Compare against at least one baseline.
- Update README with only claims supported by artifacts.

Winning acceptance criteria:

- `results/eval_summary.json` exists and includes baseline and improved-agent metrics.
- `results/reward_curve.png` exists and is generated from real logs.
- `results/baseline_vs_agent.png` exists.
- README embeds these plots.
- README describes exactly how the plots were produced.
- Any “MTTR reduction” or “reward improvement” claim points to a result file.

### 2. README Overclaims Compared To Evidence

Current state:

- README says the agent achieves “60% reduction in MTTR”.
- README mentions an “ignition point”.
- These may be true as intended story, but they are not backed by committed artifacts.

Why this matters:

Judges and mentors will punish unverifiable claims. A slightly modest but well-proven claim is stronger than a big claim with no evidence.

Required work:

- Replace unsupported claims with measured numbers.
- Add a “Results” section with table and plots.
- Add a “Reproduce Results” section.
- Link the HF Space, demo video/blog/slides, and any training logs.

Winning acceptance criteria:

- README has a table like:

| Agent | Phase 1 Score | Phase 2 Score | Phase 3 Score | Survival Rate | Mean Reward |
|---|---:|---:|---:|---:|---:|
| Random baseline | ... | ... | ... | ... | ... |
| Heuristic baseline | ... | ... | ... | ... | ... |
| Trained/improved policy | ... | ... | ... | ... | ... |

- All values in the table are generated by a script committed in the repo.
- README does not contain unsupported percentages.

### 3. Training Script Does Not Yet Train Against The Environment

Current state:

- There is a notebook with GRPO setup.
- There is synthetic SFT data.
- The reward functions in the notebook mostly reward format validity.
- `training/grpo_train.py` does not run real GRPO against environment trajectories.

Required work:

Create a real training/evaluation pipeline with these layers:

1. Environment rollout collector.
2. Baseline policies.
3. Reward logging.
4. Optional SFT/GRPO training.
5. Evaluation on held-out seeds.
6. Plot generation.

Minimum practical implementation:

- If full GRPO is too slow, implement a credible “policy improvement” pipeline:
  - Random baseline.
  - Heuristic baseline.
  - LLM baseline.
  - SFT-trained or prompt-improved agent.
  - Evaluation over the same seed set.

Winning implementation:

- Use TRL/Unsloth for actual fine-tuning.
- Use environment rollouts to generate reward-labeled samples.
- Evaluate pre-training vs post-training on held-out seeds.
- Commit the exact metrics and plots.

Suggested files:

- `training/rollout.py`
- `training/evaluate_policy.py`
- `training/train_grpo.py`
- `training/train_sft.py`
- `training/plot_results.py`
- `results/rollouts_baseline.jsonl`
- `results/rollouts_trained.jsonl`
- `results/eval_summary.json`
- `results/reward_curve.png`
- `results/baseline_vs_agent.png`

## Domain 3: Reward Design

Current strengths:

- R1-R5 reward structure exists.
- Weights are documented in `openenv.yaml`.
- Watchdog penalty exists.
- Reward breakdown is returned in `StepResult`.

Current weaknesses:

- The reward can be shallow and gameable.
- `emit_signal` can be used repeatedly and still receive some efficiency/metabolic benefit.
- R4 is mostly intent matching, not true multi-agent coordination.
- R5 depends on held-out seeds but held-out seed evaluation is not clearly demonstrated.

Required tasks:

- Add stronger no-op and repeated-signal penalties.
- Add “useful action” checks:
  - Did patch reduce failing tests?
  - Did quarantine reduce blast radius?
  - Did rollback recover from a regression?
  - Did expert request happen after a patch or ambiguous failure?
- Make R4 require actual sequence quality:
  - signal intent
  - act on same target
  - improve tests or reduce risk
- Make R5 evaluate held-out generalization:
  - separate evaluation seeds
  - explicit held-out metrics in `results/eval_summary.json`
- Add reward-hacking tests.

Suggested reward structure:

| Reward | Keep | Improve |
|---|---|---|
| R1 SLA/vitality | Yes | Tie more directly to SLO metrics, not only vitality delta. |
| R2 recovery | Yes | Count service-level recovery and root-cause fixes. |
| R3 efficiency | Yes | Penalize repeated no-op/signal farming harder. |
| R4 coordination | Yes | Require subagent usefulness or intent-action-result chain. |
| R5 generalization | Yes | Evaluate on committed held-out seeds and report separately. |

Acceptance criteria:

- A random/no-op policy scores poorly.
- A patch-spamming policy scores poorly.
- A signal-spamming policy scores poorly.
- A targeted repair policy scores clearly higher.
- Reward breakdown explains why.

Winning additions:

- Add ablation study:
  - Full reward.
  - No R4.
  - No watchdog.
  - Sparse reward only.
- Show that composable rewards improve learning stability.

## Domain 4: Training Pipeline

Current strengths:

- Notebook exists.
- Training dependencies are separated.
- SFT data exists.
- Gym wrapper exists.

Current weaknesses:

- GRPO script is not real enough.
- Notebook uses mock plots.
- Reward callbacks are not connected to actual environment outcomes.
- No clear trained model artifact.

Required tasks:

### Step 1: Implement Baselines

Add:

- random policy
- do-nothing policy
- heuristic repair policy
- LLM policy using `inference.py`

Metrics:

- survival rate
- mean episode reward
- final vitality
- tests recovered
- mean steps to recovery
- watchdog violations

### Step 2: Implement Rollout Logging

Every episode should log:

```json
{
  "episode_id": 1,
  "task_id": "phase_1",
  "seed": 123,
  "policy": "heuristic",
  "steps": 12,
  "total_reward": 2.45,
  "survived": true,
  "thrived": false,
  "final_vitality": 87.5,
  "watchdog_violations": 0,
  "actions": [...],
  "reward_history": [...]
}
```

### Step 3: Implement Real Evaluation

Run each policy over:

- 20 seeds for Phase 1.
- 20 seeds for Phase 2.
- 20 held-out seeds for Phase 3.

Minimum:

- 5 seeds per phase if time is short.

Winning:

- 50+ episodes total.
- Same seed set for baseline and improved policy.

### Step 4: Implement Training

Options, in order of credibility:

1. Full GRPO with TRL/Unsloth.
2. SFT on successful traces, then evaluate before/after.
3. Prompt policy improvement with controlled evaluations.
4. Heuristic improved policy as fallback, clearly labelled as “expert heuristic baseline”, not trained.

Do not call heuristic improvement “trained” unless a model was actually trained.

Acceptance criteria:

- Training/evaluation can be rerun by judges.
- A generated `results/eval_summary.json` is committed.
- README includes exact command:

```bash
python training/evaluate_policy.py --policy random --episodes 20 --out results/random.jsonl
python training/evaluate_policy.py --policy heuristic --episodes 20 --out results/heuristic.jsonl
python training/plot_results.py --inputs results/random.jsonl results/heuristic.jsonl
```

Winning additions:

- Trackio or W&B link.
- Hugging Face model repo link for LoRA.
- Training notebook with executed output cells.

## Domain 6: README And Storytelling

Current strengths:

- The project has a memorable narrative.
- SRE framing is compelling.
- The README is short enough to skim.

Current weaknesses:

- It overclaims.
- It does not embed proof.
- It lacks a reproduction-first structure.
- It does not clearly distinguish environment, agent, training, and results.

Required README structure:

1. One-line pitch.
2. Why this matters.
3. What the agent sees.
4. What the agent can do.
5. Reward function.
6. Results with plots.
7. How to run the HF Space.
8. How to run locally.
9. How to reproduce evaluation.
10. Links: HF Space, demo video/blog/slides, model/logs.

Suggested pitch:

“Autonomous SRE is an OpenEnv environment where an LLM learns incident response inside a self-corrupting service cluster: diagnose faults, patch services, rollback bad states, isolate failures, and preserve SLA under chaos pressure.”

Avoid:

- “Industry-grade” unless backed by stronger evidence.
- “60% MTTR reduction” unless measured.
- “100% OpenEnv compliant” unless verified.
- “GRPO-trained” unless actually trained.

Winning story arc:

1. Problem: LLMs are weak at long-horizon operational repair.
2. Environment: A service cluster breaks under chaos faults.
3. Agent: Must preserve SLA using realistic SRE tools.
4. Learning: Reward teaches repair, efficiency, coordination, generalization.
5. Evidence: Improved policy survives more episodes and recovers more tests.
6. Impact: This trains incident-response behavior, not just code completion.

## Winning Definition

This project becomes winner-grade when the following are true:

- The environment is runnable from HF Space.
- The README explains the environment in under 5 minutes.
- The environment is clearly more ambitious than toy games.
- The reward function is coherent and hard to game.
- There is real improvement evidence from actual rollouts/training.
- Plots are committed and reproducible.
- Claims are honest and traceable to result artifacts.
- The demo shows an incident, an agent action, reward feedback, and recovery.

## Final Honest Take

The core idea is good enough to compete. The implementation is good enough to build on. The current submission is not yet winner-grade because it lacks hard proof.

The path to winning is not “add more features.” The path is:

1. Prove the environment teaches something.
2. Show baseline versus improved behavior.
3. Make the evidence impossible to miss.
4. Tell the story with discipline.

If those four things are done, this project can move from “interesting prototype” to “serious finalist.”
